Current conclusion about the CHOC WebView patch
- The local CHOC safe-area patch is no longer needed.
- The real iPhone app now works on stock CHOC because the real fix lives in repo code:
  - the host grabs the native `WKWebView`
  - sets `scrollView.contentInsetAdjustmentBehavior = .never`
  - disables automatic scroll-indicator inset adjustment
- The desktop React refactor did not remove the CHOC dependency itself because desktop still uses the stock Cmajor `JUCEPluginBase -> PatchWebView -> choc::ui::WebView` path.
- So the right answer is:
  - the CHOC WebView patch is not a permanent architectural requirement
  - it is not a current implementation requirement anymore either
  - but CHOC itself is still a current runtime dependency until the remaining native WebView layer is in-housed too

What "finish the shell refactor properly" means if the goal is to remove CHOC entirely
- iPhone:
  - Replace `choc::ui::WebView` inside `PatchWebViewHost` with a repo-owned `WKWebView` wrapper.
  - Keep the existing repo-owned bridge protocol, resource resolver, and bundle/dev-server loader.
  - The safe-area behavior already lives in repo code, so this remaining step is about removing the CHOC wrapper layer itself, not about restoring the old patch.
- Desktop:
  - Replace the current `JUCEPluginBase` and `cmaj::PatchWebView` host path with a repo-owned JUCE editor and repo-owned WebView host.
  - The current desktop React UI can sit on top of that later, but it does not do that yet.

Practical rule for future sessions
- Do not say "the native-shell refactor is done everywhere" or "desktop is already on the repo-owned host".
- The accurate statement is:
  - iPhone shell refactor: mostly done, uses stock CHOC WebView, and no longer needs a CHOC fork
  - desktop React UI refactor: done for the web UI layer, not for the native host layer

OTT lab transient clamp fix on 2026-04-12
- The default OTT lab patch was creating a loud onset spike because the band input/output gain boosts were active immediately, while the compressor gain reduction started from silence and took too long to catch up. OTT lab static per-band drive on 2026-04-12
- Added a global `Band Drive` parameter to `fx/ott_lab/OttLab.cmajor`. It drives all three bands into their own soft saturator after that band's OTT compression and band output gain, before the bands are summed back together.
- This is deliberately static gain staging. There is no envelope follower, no dynamic output trim, and no automatic loudness compensation. At `Band Drive = 0%`, the saturator returns the compressed band sample unchanged.

Cosimo synth MSEG/articulation architecture stocktake on 2026-05-05
- Intent: Take stock of the current Cosimo synth modulation setup before planning the MSEG two-shape migration and per-note articulation recall.
- Current setup: Each modulation MSEG slot is one patch-level shape plus playback settings stored under `modulation.v1`. The UI renders one 2051-sample buffer per slot and uploads it to Cmajor. Cmajor's `MsegReader` samples one buffer per slot; there is no Shape A, Shape B, morph value, keyswitch system, articulation recall system, or SysEx articulation parser yet.
- Decisions: Future MSEG slots should hold Shape A, Shape B, and an automatable realtime morph value. Shape A and Shape B are patch-level design data. Morph is recallable and automatable, and morph changes must affect currently held notes immediately. Keyswitches and Future DAW SysEx articulation messages both change only the next note, not voices already sounding.
- Decisions: Future DAW's current private SysEx contract should be used as the note-start articulation selector. Cosimo will interpret `selectorA` as a plugin-owned complete articulation snapshot selector, meaning each selected articulation carries every recallable parameter value rather than sparse overrides. `selectorB` is reserved for a later secondary articulation specialization and should not drive behavior yet.
- Findings: Future DAW's implemented v1 note-meta SysEx is a 16-byte private message emitted immediately before the note-on it belongs to. It carries channel, note number, `selectorA`, `selectorB`, duration samples, and age samples. Product paths currently fill `selectorA` from the sound articulation manifest and force `selectorB` to zero/reserved.
- Decisions: Complete articulation snapshots should include bus-level effect parameters such as distortion and chorus. It is acceptable that recalling those bus-level values changes already-held notes, because those processors currently sit after the mixed voice engine.
- Decisions: The plugin UI should expose visible articulation-slot buttons that choose which complete articulation snapshot the user is editing. This UI edit selection is separate from playback selection. Keyswitches and Future DAW SysEx should not change the visible UI edit slot by default; the user changes the UI slot manually. A future `ui follows playback` setting may opt into making the UI switch to the selector chosen by keyswitch/SysEx, matching the audio-side playback selection.
- Decisions: Editing or manually selecting a visible articulation slot should make the audible synth follow that visible slot immediately, so editing feels direct. A later keyswitch or Future DAW SysEx selector should make playback jump back to the MIDI-selected articulation. The current audible articulation selector is therefore last-writer-wins between manual UI slot selection and incoming MIDI selectors.
- Decisions: Articulations should behave like a dynamic list, not a fixed visible bank. The UI should show only defined articulation slots plus an `add` control, so it feels like users can add as many as they need. Internally the system can enforce an appropriate cap; Future DAW's current `selectorA` transport is 7-bit, so the wire format cannot address more than 128 selector values.
- Decisions: Cosimo should mirror Future DAW's articulation model by storing a stable numeric `selectorA` on each articulation plus UI-facing fields such as name/label. `selectorA` should not be an implied list index, because deleting or reordering articulations must not change what existing MIDI/SysEx numbers mean. Missing selector values should be ignored, leaving the current audible articulation unchanged rather than falling back to default.
- Direction: When Cosimo eventually runs as an integrated internal synth inside Future DAW, Future DAW's sound manifest should be the full source of truth for the articulation selector/name map and should dictate what Cosimo loads and what Future DAW displays. During standalone plugin development, Cosimo still needs a local patch-owned articulation state, but that state should be shaped so it can be supplied by Future DAW later rather than becoming a conflicting second model.
- Decisions: Approved working assumption: Future DAW's sound manifest owns articulation metadata and stable selectors (`id`, label/icon/order/default, `selectorA`, and eventually `selectorB`). Cosimo owns a separate versioned articulation bank payload containing the complete parameter snapshots those selectors point at. In standalone Cosimo this payload can live in patch stored state. In integrated Future DAW it can be attached to the sound/package as embedded JSON or a referenced file; that physical packaging choice is deferred.
- Decisions: The first implementation direction should focus on the MSEG Shape A/B plus realtime morph migration before building the larger articulation-bank machinery.
- Decisions: The MSEG Shape A/B editor should show both shapes together for comparison. The selected shape is editable with normal control points; the other shape is drawn as a muted reference. Explicit A/B edit buttons choose which shape is edited, and the morph control remains visible as the realtime/automatable blend value.
- Decisions: Selecting Shape A or Shape B for editing should not change the audible morph position. The current morph value remains the audio source of truth. Shape edit focus only determines which control points are editable; optional A/B audition controls can be considered later but are not part of the first implementation.
- Decisions: The MSEG v2 state shape should store `shapeA`, `shapeB`, `morph`, and `playback` per MSEG slot. Runtime upload should keep one structured MSEG buffer endpoint and add a `shapeIndex` field, rather than creating separate A/B endpoint names. Playback upload remains separate. Morph should be a visible automatable Cmajor parameter per slot, likely `mseg1Morph`, `mseg2Morph`, and `mseg3Morph`.
- Decisions: Use a hard cut for the MSEG A/B migration. Cosimo has not shipped, so do not add backwards-compatibility complexity for old saved `modulation.v1` shapes. Missing or malformed state may still fall back to new defaults, but the implementation should not carry a legacy compatibility layer.
- Decisions: Use `modulation.v2` as the stored-state key for the MSEG A/B hard cut. This is a small explicit schema break and avoids interpreting stale `modulation.v1` development state as valid.
- Open Questions: Define the first implementation/test sequence.
